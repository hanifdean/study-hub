Chapter 2: Variations of Program Theory Over Time
What is Program Theory?

Program theory is an explicit description of how an intervention is expected to produce its outcomes — the causal chain from what you do to what you hope to achieve. It answers: "Why do we think doing X will lead to result Y?"

Despite being called "new," the roots go back 50+ years. Key ideas have been articulated, forgotten, and rediscovered repeatedly.

The First Boom (1980s)

Driven by public sector reforms demanding managing for results, not just compliance:

  • Chen & Rossi — "Theory-driven evaluation": six domain theories (treatment, implementation, environment, outcome, impact, intervening mechanisms, generalization)
  • Bickman — Special issues on program theory in evaluation journals (1987, 1990)
  • Australia, Canada — governments introduced program management requiring departments to articulate goals, explain plans, and report on performance
  • New South Wales — Bryan Lenne pioneered "outcomes hierarchies" where external factors were included in program theory
Two Competing Representation Styles
Style 1: Standardized Template

A fixed number of boxes (usually 4-5) in a linear pipeline: Inputs → Activities → Outputs → Outcomes → Impact

Examples: Logframe, United Way model, W.K. Kellogg Foundation Logic Model

Pro: Easy to communicate, consistent across programs. Con: Oversimplifies, forces everything into predetermined categories

Style 2: Flexible Chain of Results

Multiple branching pathways with many intermediate outcomes — no fixed number of levels or boxes

Examples: Weiss's causal models, outcomes hierarchies

Pro: Captures complexity and multiple causal pathways. Con: Harder to standardize, can become unwieldy

Both styles continue to be used. The choice depends on purpose: standardized templates for reporting and accountability; flexible chains for understanding and learning.

Key Innovations
Realist Evaluation (Pawson & Tilley, 1994/1997)

Program theory as Context-Mechanism-Outcome (CMO) configurations. The critical insight: causal mechanisms only fire in favorable contexts. A program doesn't "work" universally — it works for particular people in particular settings.

Outcome Mapping (Earl, Carden & Smutylo, 2001)

For interventions where implementers can't control impacts but seek to influence them through "boundary partners." Acknowledges that many public programs work indirectly — changing others' behavior to produce outcomes, not producing outcomes directly.

⚡ Key insight: Program theory is used for BOTH planning (designing programs) AND evaluation (assessing programs). Kirkpatrick said: first use the chain of objectives to plan the program (work backwards from desired outcomes), then use the same chain as a framework for evaluation.

Alternative Conceptualization (Zimmerman/Stacey)

A 2x2 matrix of agreement about ends × certainty about means:

  • Simple: Agreement about ends AND certainty about means
  • Technically complicated: Agreement about ends, uncertainty about means
  • Socially complicated: Disagreement about ends, certainty about means
  • Complex: Neither agreement nor certainty
Snowden's Cynefin Framework
  • Simple = domain of the known; cause and effect well understood; best practices apply
  • Complicated = domain of the knowable; expert knowledge required
  • Complex = domain of the unknowable; patterns only visible in retrospect
  • Chaotic = no patterns even in retrospect
  • Disordered = classification unclear
Implications for Program Theory
Simple Interventions

Discrete, standardized, one agency, works the same everywhere. Both necessary and sufficient for outcomes. Constant dose-response. Can be evaluated as "what works."

Few real interventions meet these criteria exactly.

Complicated Interventions — Multiple Components

Different valued outcomes: Partner organizations value different impacts (e.g., employment dept values jobs; education dept values qualifications; local gov values completed projects). These can conflict.

Multiple and competing objectives: A participant who leaves a program for a job = success for employment, failure for training completion.

Multiple levels: Interventions operate at individual, organizational, and system levels simultaneously.

Multiple implementing agencies: Different organizations with different cultures, accountabilities, and definitions of success.

Complex Interventions — Adaptation & Emergence

Evolving focus: Objectives and activities change as the program develops.

Emergent outcomes: Results that weren't planned or predicted.

Fluid boundaries: Where the program ends and the context begins is unclear.

Feedback loops: Outcomes influence inputs; the system responds to its own outputs.

⚡ Critical point: Most interventions have elements of ALL three. Vaccination is simple (one shot = immunity), complicated (logistics across a country), and complex (persuading people to participate). The classification is a heuristic, not an absolute label. The question is: which aspects of your intervention need which type of thinking?

Chapter 10: Critiquing Program Theory

The goal of a critique is to ensure the program theory is useful rather than perfect. Program theories evolve over time — a review checks whether the current version is fit for purpose.

Program Theory "Traps" (As Positive Questions)
✅ Checklist for Sound Program Theory

• Does it show logical and defensible relationships between what the program does and the outcomes it seeks?

• Does it present a plausible solution to an identified problem?

• Are intended outcomes specified clearly enough to determine whether the approach is relevant?

• Could achievement of outcomes be recognized, if not precisely measured?

• Does it consider unintended outcomes?

• If complicated or complex: does it include coordination mechanisms, feedback loops, emergent outcomes, fluid boundaries?

• Can it generate useful evaluation questions?

• Is the type of program theory fit for its audience and purpose?

Two Sets of Criteria
Internal Validity — Does It Hang Together?

Clarity of description: Enough detail to see how the program will work — problem description, intended outcomes, planned activities, resources, and external factors.

Outcomes chain as central organizer: Many program theories fail by concentrating on what the program does rather than what it achieves. Outcomes should range from immediate → intermediate → ultimate, arranged in sequence.

Outcomes relate to the problem: Do intermediate outcomes mirror the causes the program addresses? If the problem is "parents don't encourage reading," there should be an outcome about "parents become more supportive of reading."

Strength of logical argument: Test from both directions — from the end ("what needs to happen to achieve this?") AND from the start ("if we do this, what happens next?"). Check for gaps, breaks in temporal sequencing, and whether the program is actually sufficient.

External Validation — Is It Supported?

Evidence base: Is there research supporting the causal claims? Is the evidence current?

Consistency with accepted theories: Does the program theory align with what we know from social science, behavioral science, etc.?

Context considered: Does it account for how context affects outcomes? (Connects to Pawson & Tilley's CMO configurations)

Ethical base: Is the program theory ethically sound?

Worth the effort: Given the purposes, was developing this program theory a good use of time and resources?

Key Distinctions
⚡ Inputs, outputs, and outcomes are RELATIVE: What's an outcome for one program is an input for another. An advocacy program's outcome (funds generated) is the input for a service delivery program. A team-building activity's outcome (partners working together) looks like an activity — but if your program theory is about building the partnership, it IS an outcome. Context determines classification.
Diagnosing Program Failure

When a program fails, a critique can help identify whether the failure was due to:

  1. Theory failure — the program theory itself was wrong (doing X doesn't actually lead to Y)
  2. Implementation failure — the theory was sound but wasn't implemented properly
  3. Context failure — the theory was sound, implementation was good, but the enabling context wasn't present

This three-way distinction is crucial — without it, good programs get abandoned because of bad implementation, or bad theories persist because they were "well implemented."

💡 Key Takeaway

Every public program rests on a theory — an assumption about why doing X will produce result Y. Making that theory explicit is the first step to both good design and good evaluation. But the theory must match the nature of the intervention: simple theories for simple programs, theories that embrace multiple components for complicated ones, and theories that allow for emergence and adaptation for complex ones. "As simple as possible, but no simpler."
