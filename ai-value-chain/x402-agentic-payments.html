üö´ The Problem: Internet Payments Are Broken

Today‚Äôs payment rails were built for humans, not software agents. Credit cards and subscription billing are high-friction, expensive, and unprogrammable at scale.

High friction

Accounts, KYC, and API keys create weeks-long onboarding just to pay for a tiny API call.

Bad economics

Card fees + chargebacks make micro-payments uneconomical. Minimums force ‚Äúall you can eat‚Äù pricing.

Not programmable

Agents can‚Äôt autonomously pay or settle. No native mechanism to pay per request in-band with HTTP.

Opaque access

Usage caps, credits, and quotas live outside the request itself. No standard price discovery.

üìú HTTP 402 ‚Äî The Status Code That Never Got Used

HTTP 402 (Payment Required) has existed since HTTP/1.1, but it was never standardized. The result: no canonical way for servers to say ‚Äúpay me for this resource‚Äù and no universal client behavior for handling it.

x402 fills that missing protocol gap: it standardizes how a server responds with payment requirements and how a client supplies payment proof ‚Äî making pay-per-request a first-class HTTP primitive.

üß© Key Terms & Roles

Each request involves a resource server, a 402 response, and a facilitator to verify the payment proof.

Resource

The content or API endpoint being requested (data, inference, tool access).

Client

The agent or app making the HTTP call and paying for access.

Resource Server

The server that owns the resource and enforces payment requirements.

Facilitator

A payment service that helps verify and settle the on-chain payment proof.

üß∞ Integration: 1-Line Server Middleware + 1-Function Client

The core idea: x402 feels like HTTP middleware, not a payment product. SDKs exist for TypeScript, Python, and Go.

Server (Express / Next.js / Hono)

Client (fetch wrapper)
